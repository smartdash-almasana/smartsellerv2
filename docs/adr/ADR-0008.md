# ADR-000X — Canonical Rebuild Strategy: V2 Stabilization + Parallel V3

- **Status:** Proposed
- **Date:** 2026-03-10
- **Owners:** SmartSeller Architecture
- **Decision Type:** Structural / Irreversible without superseding ADR

## Context

SmartSeller is a clinical operational risk system. Its core purpose is not reporting but deterministic detection of operational, reputational, and financial damage before it formalizes.

The current V2 system has achieved partial operational recovery, but the architecture shows structural signals that make indefinite continuation by patching increasingly expensive and risky:

- multiple active writers in critical pipeline stages;
- historical overlap between legacy and canonical flows;
- repeated need for forensic validation to confirm state propagation;
- growing cognitive cost to explain and validate end-to-end behavior;
- risk that future integrations, especially Shopify, adapt the core in the wrong direction.

At the same time, V2 is not fully disposable:
- there is live operational value;
- critical pipeline segments have been validated and recovered;
- production continuity must not be jeopardized by a big-bang rewrite.

## Decision

SmartSeller will **not** use V2 as the long-term structural base of the platform.

SmartSeller will adopt a **dual-track strategy**:

1. **V2 remains in operational stabilization mode**
   - only critical fixes;
   - only integrity, reliability, and validation work;
   - no major expansion of domain scope on top of V2.

2. **A parallel V3 canonical core will be opened**
   - designed from first principles;
   - provider-agnostic at the core;
   - deterministic, idempotent, traceable, and multi-tenant safe;
   - migrated progressively by pipeline stage, without big-bang replacement.

## Rationale

This decision is taken because continued patching of V2 is still useful for short-term continuity, but no longer sufficient as a long-term architecture strategy.

A full immediate rewrite is also rejected because it would create unnecessary delivery and migration risk.

A parallel canonical rebuild provides the best balance between:
- operational continuity;
- architectural correction;
- controlled migration;
- future scalability.

## Scope of V3

V3 will be built around the canonical pipeline:

`webhook_events -> domain_events -> snapshots -> metrics -> clinical_signals -> health_score`

The V3 core must preserve these invariants:

- strict multi-tenant identity:
  - `tenant_id`
  - `store_id`
  - `seller_uuid`
  - `external_id`
  - `provider_key`
- deterministic state reconstruction;
- one governed writer per stage;
- no provider-specific fields in core tables;
- idempotent materialization by deterministic keys;
- reproducible score with no external runtime dependency.

## Migration Policy

Migration will be progressive, not big bang.

Recommended order:

1. identity and tenancy
2. ingest layer
3. snapshots
4. metrics
5. clinical signals
6. health scores
7. provider-specific extensions

Each stage may move only when:
- schema is approved;
- invariants are explicit;
- validation path is defined;
- rollback or coexistence is understood.

## Constraints

The following rules apply immediately:

### V2
Allowed:
- critical production fixes;
- integrity corrections;
- reconciliation hardening;
- operational validation;
- minimal documentation updates.

Not allowed:
- structural expansion as if V2 were canonical;
- new domain complexity without ADR;
- provider-driven distortion of the core.

### V3
Required:
- ADR-backed irreversible decisions;
- explicit ownership of each writer;
- clinical traceability from score to snapshot to event to run;
- provider adapters separated from core.

## Consequences

### Positive
- lowers long-term architectural risk;
- prevents Shopify from shaping the core incorrectly;
- reduces future QA and forensic burden;
- creates a clean foundation for deterministic clinical scoring.

### Negative
- temporary dual maintenance cost;
- more upfront design discipline required;
- migration governance becomes mandatory.

## Go / No-Go Criteria

This ADR is justified because the following conditions are already present in V2:

- critical stages have or had more than one writer;
- validation cost is forensic rather than routine;
- structural ambiguity exists between legacy and active flows;
- future integrations risk inheriting accidental complexity.

If these conditions materially disappear, this ADR may be revisited by a superseding ADR.

## Outcome

**Approved direction:**  
Open **V3 canonical core** in parallel.  
Keep **V2** as a temporary operational stabilization layer only.
